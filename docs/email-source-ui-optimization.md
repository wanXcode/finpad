# FinPad 邮箱数据源 UI 优化方案 v1.2

> 版本：v1.2 | 日期：2026-03-17 | 变更：新增数据模型重构 + 同步详情可查看

---

## 一、背景与目标

当前"数据导入 → 邮箱抓取"功能已具备基础能力，但存在以下问题：

**目标**：让用户顺畅完成「配一个邮箱 → 勾选多个平台 → 触发同步 → 处理加密附件 → 看到入库结果」的完整闭环。

---

## 二、当前问题

### P0 Bug：前后端 config_json 字段名不一致
前端写 `imap_email`/`imap_password`，后端读 `email`/`password`。**（已修复 ✅）**

### P0：数据模型不合理（v1.2 新增）

**现状**：一个数据源 = 一个邮箱 + 一个平台。用户如果同一个邮箱收支付宝和微信两种账单，需要创建两个数据源，邮箱配置（IMAP 地址、邮箱、密码）填两遍。

**应该是**：一个邮箱配置一次，然后勾选这个邮箱里有哪些平台的账单（支付宝、微信、招行、工行等）。

```
当前（错误）：
  数据源1: 邮箱A + 支付宝
  数据源2: 邮箱A + 微信    ← 邮箱配置重复

应该（正确）：
  邮箱通道: 邮箱A
    ├── ✅ 支付宝
    ├── ✅ 微信支付
    ├── ☐ 招商银行
    └── ☐ 工商银行
```

### P0：加密 ZIP 附件无法处理
支付宝、招行每次 ZIP 密码不同，不能预设。**（已实现后端 pending 机制 ✅，前端 UI 已上 ✅）**

### P1：同步状态不可交互（v1.2 新增）

**现状**："⚠ 部分成功"只是一个 badge，不能点击，用户不知道什么成功了、什么没成功。

**应该**：点击状态 badge 或卡片可以展开同步详情，显示每个平台的同步结果。

### P1：配置不可查看/编辑
**（已实现 ✅）**

### P2：同步状态信息不够
**（已实现 ✅ — 6 种状态）**

## 三、v1.2 优化方案

### 3.1 数据模型重构：邮箱通道 → 多平台（P0）

#### 核心思路

把"数据源"的概念从「邮箱+平台」改为「邮箱通道」，一个邮箱通道可以关联多个账单平台。

#### 数据库变更

**方案 A（简单，推荐）**：保持 `data_sources` 表结构不变，`platform` 字段改为存多选（JSON 数组）

```
data_sources.platform: "alipay"  →  '["alipay","wechat"]'
```

- 最小改动，前后端适配简单
- config_json 中的邮箱配置只存一份

**方案 B（规范）**：拆表

```sql
-- 邮箱通道（一个邮箱一条记录）
CREATE TABLE email_channels (
    id, user_id, name, imap_host, email, password,
    sync_interval_minutes, enabled, ...
);

-- 邮箱通道 × 平台（多对多）
CREATE TABLE channel_platforms (
    id, channel_id, platform, enabled, ...
);
```

- 更规范，但改动大，涉及 API、同步引擎、前端全链路

**建议用方案 A**：简单直接，后续有需要再拆。

#### 前端交互变更

**添加邮箱通道**（新流程）：

```
┌──────────────────────────────────────────────┐
│ 添加邮箱通道                                  │
│                                              │
│ 通道名称    [我的126邮箱                   ]   │
│ 邮箱服务商  [▼ 126 邮箱                    ]   │
│ 邮箱地址    [airname@126.com               ]   │
│ IMAP 授权码 [••••••••                 👁   ]   │
│ 同步间隔    [▼ 每 10 分钟                  ]   │
│                                              │
│ 开通平台（勾选邮箱中接收的账单类型）：          │
│ ┌──────────────────────────────────────┐      │
│ │ ☑ 💰 支付宝                          │      │
│ │ ☑ 💚 微信支付                        │      │
│ │ ☐ 🏦 招商银行                        │      │
│ │ ☐ 🏛️ 工商银行                        │      │
│ └──────────────────────────────────────┘      │
│                                              │
│              [取消]  [保存]                    │

```

**列表卡片**（新样式）：

```
┌──────────────────────────────────────────────────────┐
│ 📧 我的126邮箱                      [编辑] [▶同步] [🗑]  │
│    airname@126.com · 126 邮箱 · 每 10 分钟                │

│                                                          │
│  同步结果：                                               │

│  │ 💰 支付宝  ✓ 成功 · 新增 14 条               │        │

│  └──────────────────────────────────────────────┘        │
└──────────────────────────────────────────────────────────┘
```

#### 后端变更

**同步逻辑**：`sync_from_email` 接收 platforms 数组，一次 IMAP 连接中按多个平台的关键词搜索邮件，分别处理。

**API**：
- `POST /api/sources`：`platform` 字段改为接收数组 `["alipay", "wechat"]`
- `PATCH /api/sources/{id}`：同上
- `POST /api/sources/{id}/sync`：同步时按 platforms 逐个处理，返回每个平台的结果
- 同步结果 response 改为：

```json
{
  "status": "partial",
  "results": {
    "alipay": { "status": "success", "records_created": 14, "records_skipped": 108 },
    "wechat": { "status": "pending_password", "pending_count": 1 }
  },
  "message": "支付宝同步成功，微信有1个加密文件待输入密码"
}
```

#### 数据迁移

现有数据（同一邮箱的多条记录）需要合并：
1. 找到 email 相同的 data_sources 记录
2. 合并 platform 为数组
3. 删除重复记录，保留一条

当前只有 2 条（支付宝和微信，同一邮箱），可以手动迁移或写简单脚本。

### 3.2 同步详情可查看（P1，v1.2 新增）

#### 问题

状态 badge 只是一个标签，用户看到"部分成功"但不知道具体哪个平台成功、哪个失败。

#### 方案

**点击卡片或状态 badge → 展开同步详情区域**：

```
同步详情（上次：2026-03-17 12:00）
┌──────────────────────────────────────────────┐
│ 💰 支付宝  ✓ 成功                            │
│    新增 14 条，跳过 108 条（重复）             │
│                                              │
│ 💚 微信支付  ⏳ 待输入密码                    │
│    📎 微信支付账单(20260206-0306).zip          │
│    [________密码________] [解锁并导入]         │
└──────────────────────────────────────────────┘
```

- 每个平台一行，显示独立的状态和结果
- pending 的文件在对应平台下方显示密码输入框
- 失败的平台显示错误原因

#### 数据存储

`last_sync_message` 改为存储 JSON：

```json
{
  "alipay": { "status": "success", "records_created": 14, "records_skipped": 108 },
  "wechat": { "status": "pending_password", "pending_count": 1 }
}
```

前端解析 JSON 后按平台展示详情。如果 JSON 解析失败（旧数据），fallback 显示纯文本。

| 操作 | 说明 |
|------|------|
